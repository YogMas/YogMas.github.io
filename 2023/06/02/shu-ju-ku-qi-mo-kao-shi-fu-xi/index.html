# 第二章 关系及其相关概念 (关系实体)

关系模型 概念 关系运算

  1. 相关概念
    • 实体：包含有数据特征的事物对象在概念模型世界中的抽象名称
    • 关系：具有关系特征，用于存放实体数据的二维表
    • 键：唯一标识元组的属性列
    • 复合键：唯一表示元组的多列作为键
    • 候选键：多个列适合作为建，其中每个都称为候选键
    • 主键：最具有代表性的一个候选键
    • 代理键：采用 DBMS 自动生成的序列作为主键（比复合主键拥有更高的效率）
  2. 主键的作用
    • 唯一表示元组 z
    • 与外键建立联系，实现关系表之间的连接
    • 数据库文件使用主键值来组织关系表的数据存储
    • 数据库使用主键索引快速检索数据

# 关系模型原理

关系模型：一种基于二维表结构存储数据实体及实体间联系的数据模型

关系运算：常见的包括 选择，投影，连接，除，并，交，差

数据关系约束:

  • 实体完整性约束
  • 参照完整性约束
  • 用户自定义完整性约束

关系模型数据操作

完整性约束的作用 (不大可能考)

  • 消除关系表的元组重复存储 (对应实体完整性)
  • 保持关联表的数据一致性 (对应参照完整性)
  • 实现业务数据规则 (对应用户自定义完整性)

Null 值：代码该属性可以没有具体值

  1. 实体完整性约束：在关系表中实施的主键取值约束，以保证关系表中的每个元组可以被唯一标识

    • 每个关系表中的主键属性列不允许为空值
    • 现实世界中的实体都是靠主键来标识， 主键取值应唯一

  2. 参照完整性：关系表之间需要遵守的数据约束，以保证关系之间关联列的数据一致性

  3. 参照完整性约束规则：若关系表 R 中的外键 F 和关系 S 中的主键 K 相关联，则 R 中外键 F 值必须与 S 中主键 K 值一致 (外键取值必须与现有主键值对应)

  4. 外键：在关联的两个关系中，它们具有一个或多个相同属性，若关联列在第一个关系中作为主键，则在第二个关系中作为外键

  5. 以下情况会带来关联表的参照完整性问题

    用户自定义完整性：用户根据具体业务对数据处理规则要求所定义的数据约束 (可以通过建表语句来记忆)

    • 定义列的数据类型与取值范围
    • 定义列的缺省值
    • 定义列是否允许取空值
    • 定义列取值唯一性
    • 定义列之间的数据依赖性

# 索引

索引：一种按照关系表中指定的取值顺序组织元组数据存储的数据结构，使用他可以加快表中数据的查询访问

索引作用：支持对数据库表中数据快速查找

优点:

  • 提高数据检索速度
  • 快速连接关联表
  • 减少分组和排序时间

开销：

  • 占用额外空间
  • 维护索引带来系统性能开销

# 视图

视图：一种通过基础表或其他视图构建的虚拟表，自身没有自身的数据，而是使用了存储在基础表中的数据 (实际存储的是查询语句)

优点:

  • 提高数据访问安全性：可以将数据库中敏感数据隐藏起来，外部用户无法得知数据表的完整数据
  • 提供一定程度的数据逻辑独立性：数据表结构发生改变，只要视图结构不变，应用程序可以不作更改
  • 集中展示用户所感兴趣的特定数据：可以将用户不关心的数据进行过滤

# 第四章 数据库设计

  2. 数据库结构模型设计 - 数据库结构模型 (大概率不考)

    • 概念数据模型 Conceptual Data Model (CDM): 从用户角度所建模的系统数据对象及其关系，它帮助用户分析信息系统的数据结构关系。
    • 逻辑数据模型 Logic Data Mode (LDM): 从系统分析员角度所建模的系统数据对象逻辑结构关系，它帮助开发人员分析信息系统的逻辑数据结构。
    • 物理数据模型 Physical Data Model (PDM): 从系统设计人员角度所建模的系统数据物理存储及结构关系，它针对设计者具体定义信息系统的数据库表结构。

  3. E-R:“实体 - 联系模型”（Entity-Relationship Model）的简称。它是一种描述现实世界概念数据模型、逻辑数据模型的有效方法，基本元素有实体、属性、标识符和联系

    • 实体：客观事物在逻辑层面的数据抽象，用于描述事物的数据对象
    • 属性：描述特征的数据项
    • 标识符：表示不同实体实例的属性
    • 联系：实体之间的关联关系
    • 元：实体之间关联的数目被称为元
    • 基数：实体之间关系存在的数量对应关系

# 第五章 数据库管理

  1. 数据库管理：是指为保证数据库系统的正常运行和服务质量必须进行的系统管理工作 (大概率不考)

  2. 事务：是指由单个业务处理单元的一组数据库访问操作，要求他们要么都成功执行，要么都不执行

  3. 事务 ACID 特性:

    • 原子性 (Atomicity): 事务所有操作在数据库中要么全部执行，要么全部不执行
    • 一致性 (Consistency): 事务执行结果使数据库从一种正确状态变迁到另一种正确状态
    • 隔离性 (Isolation): 事务与事务之间隔离，并发执行透明
    • 持续性 (Durability): 事务完成后，数据改变必须是永远的

# ODBC (概念必考一个)

出现原因：网络环境下，应用程序需要对多种数据库实现互连和资源共享，需要一种能实现连接不同 DBMS 的方法

ODBC: 实现了应用程序对多种不同 DBMS 数据库的访问，实现了数据库连接方式的变革

ODBC 定义了一套基于 SQL 的，公共的，与数据库无关的 API, 每个应用程序使用相同的源代码就可访问不同的数据库系统，存取多个数据库中的数据

ODBC 使应用程序与 DBMS 之间在逻辑上的独立性，是应用程序具有数据库无关性

ODBC 访问数据库的步骤

  1. 首先用 ODBC 管理器注册一个数据源
  2. 管理器根据数据源提供的数据库位置，数据库类型以及 ODBC 驱动程序等信息，建立起 ODBC 与具体数据库的联系
  3. 应用程序只需将数据源名提供给 ODBC,ODBC 就能建立起与相应数据库的连接
  4. 应用程序可以通过驱动程序管理器与数据库交换信息
  5. 驱动程序管理器负责将应用程序对 ODBC API 的调用传递给正确的驱动程序
  6. 驱动程序执行完相应的 SQL 操作后，将结果通过驱动程序管理器返回给应用程序

# 第七章 NoSQL 数据库

cap base 5v 四种 nosql 数据库类型和分类比较

在分布式的环境下设计和部署系统时，有 3 个核心的需求：CAP 对应一致性（Consistency），可用性（Availability）和分区容忍性（Partition Tolerance）

  • 大数据的 5V 特征：超量 Volume, 高速 Velocity, 异构 Variety, 真实 Veracity, 价值 Value

  • Cap 理论

    • Consistency 一致性：在执行某项数据的修改操作之后，所有结点在同一时间具有相同的数据，系统具有一致性
    • Available 可用性：在每一个操作之后，无论成功或者失败，系统都要在一定时间内返回结果，保证每个要求不管成功或者失败都有响应
    • Partition Tolerance 分区容忍性：系统中任意信息的丢失或者失败都不会影响系统的继续运行
    • 理论核心: CA (单点集群) CP AP

  • CAP 目的：

    • CAP 是为了探索不同应用的一致性 C 与可用性 A 之间的平衡，
    • 在网络或其他原因，通过牺牲一定的一致性 C 来获得更好的性能与扩展性
    • 在有分隔发生，选择可用性 A，集中关注分隔的恢复，需要分隔前、中、后期的处理策略， 及合适的补偿处理机制。
    • 选择什么样的方式： 放弃Ｐ？放弃Ａ？放弃Ｃ？BASE

  • BASE 理论

    • 基本可用，系统能够基本运行，一直提供服务
    • 软状态 无连接 系统不要求一直保持强一致状态
    • 最终一致性 系统在某个时刻达到最终一致性

  • BASE 特点：

    • ACID 是事物的特征， A（原子性）C（一致性）I（隔离性）D（持久性），ACID 的特点是强一致性、隔离性、采用悲观保守方法、难以变化；
    • BASE 的特点是弱一致性、可用性优先、采用乐观方法、适应变化并且简单快捷。
    • 对数据不断增长的系统， 大数据环境下系统的可用性及分隔容忍性的要求要高于强一致性，很难满足事务要求的 ACID 特性。

  • 四种不同数据库:

    • 列存储数据库，将同一列的数据存储在一起，可以存储结构化和半结构化数据
    • 键值存储数据库，存储的数据是有键（key）和值（value）两部分组成，通过 key 快速查询到其 value，value 的格式可以根据具体应用来确定
    • 文档存储数据库，存储的内容是文档型的，可以用格式化文件（类似 json、XML 等）的格式存储
    • 图存储数据库，数据以有向加权图方式进行存储
