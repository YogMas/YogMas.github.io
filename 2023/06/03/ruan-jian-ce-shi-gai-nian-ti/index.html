# 软件测试

# 引论

  1. 为什么要进行软件测试

    软件中存在的缺陷可能会给我们带来巨大损失或灾难，在推出产品和系统前，要进行软件测试

    测试是软件开发的重要组成部分

    只有经过大量的测试才能发现软件缺陷，只有发现软件缺陷，才能保证软件质量

  2. 软件测试：是一种以发现程序错误，衡量软件质量为目的，并对其是否满足用户需求进行评估的活动过程

    正向思维：验证软件正常工作

    逆向思维：找出软件有缺陷的证据

  3. 软件测试的法则

    法则 1：对于任何系统，穷尽测试是不可能的。
    法则 2：测试工作可以具有创造性，但较困难。
    法则 3：测试旨在减少系统中缺陷数目，不太可能完全消除系统所有缺陷。
    法则 4：测试是有风险的、成本高昂的。
    法则 5：测试需要有计划性，否则无法确保测试项目顺利完成。
    法则 6：测试需要有独立性，否则难以让人信服。

  4. 软件质量保证 (Software Quality Assurance, SQA): 是指通过对软件产品进行有计划的评审与审计，来确保软件开发按照产品质量过程标准实施项目的管理活动。

  5. 软件质量保证的主要任务

    对软件过程各个阶段任务的完成质量及出现问题进行评审、跟踪。
    审查和验证软件产品开发过程是否遵守适用的标准、 规程和要求，并最终确保产品满足用户需求。
    建立软件质量要素的度量机制，对软件开发的各种质量指标进行量化，为管理者提供软件开发的质量分析数据。

  6. SQL 的主要工作活动

    ・质量规范制定
    ・技术评审实施
    ・软件测试流程追踪
    ・质量标准的监督执行
    ・软件质量要素度量
    ・质量数据分析与报告

  7. SQA 与软件测试的关系

    SQA 指导软件测试的计划与执行，监督测试工作结果的客观性、准确性与有收性，并协助软件测试的工作流程改进。
    软件测试是 SOA 工作落实的重要手段，它为 SQA 提供所需的质量数据，作为次件质量评价的客观依据。
    SQA 是一项软件质量管理性工作，侧重于对软件开发流程进行评审与监控。
    软件测试是一项技术性工作，侧重于对软件质量特性进行检测与验证。

  8. 测试驱动开发 TDD:

    测试驱动开发 (Test-Driven Deve lopment，简称 TDD） 是一种不同于传统软件开发流程的过程模型。它要求在编写某个功能的代码之前先编写测试代码，然后编写功能代码，通过测试来推动整个软件开发工作的进行。

# 基本概念

  1. 缺陷：欠缺或者不完备的地方，任何违背了质量要求，违背了客户的意愿，不能满足用户的要求，都可以认为是缺陷

  2. 软件质量：软件产品或服务满足用户需求的程度

  3. 软件质量属性

    • 功能性
    • 可靠性
    • 易用性
    • 效率
    • 可移植性
    • 可维护性
    • 兼容性
    • 可扩展性

  4. 软件质量分为使用质量，外部质量，内部质量，过程质量

    使用质量:

    • 有效性
    • 生产率
    • 安全性
    • 满意度

    外部质量

    • 功能性
    • 可靠性
    • 易用性
    • 效率
    • 维护性
    • 可移植性

    内部质量

    • 代码耦合度
    • 数据耦合性
    • 程序规范性
    • 需求可溯性
    • 代码复杂度

  5. 软件缺陷：软件中存在的任何一种破坏正常运行能力的问题等，其结果导致软件产品在一定程度上不能满足用户的需要

  6. 软件缺陷常见现象

  7. 为什么会出现软件缺陷

    • 技术问题
    • 软件本身
    • 团队工作

  8. 需求阶段出现软件缺陷数目最多，由于系统分析人员于用户之间沟通不畅，用户需求经常更改，规格说明书内容不够确切

  9. 不同阶段修复软件缺陷的成本代价

    • 软件缺陷在开发早起出现的概率大，在早起进行缺陷修复的成本低
    • 软件在开发后期出现的缺陷概率小，但修复成本高
    • 软件测试工作应尽早开展

  10. 静态测试和动态测试

  11. 软件验证和软件确认:

    • 软件验证是值开发软件过程中，检验软件
